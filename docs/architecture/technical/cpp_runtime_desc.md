# C++ 运行层说明

本文描述 **仿控制器运行层（C++ 实现）** 的能力边界、与 **组态工具** 及 **现有平台（Rust + Electron + Vue）** 的协作关系，以及 **运行期对外暴露功能块数据** 时应满足的集成要点。详细协议与字段级定义宜在 `docs/api/`、`docs/specs/technical/` 中另文展开。

**关联文档**：[类 OpenPLC Editor 桌面组态工具 — 架构设想](./openplc_style_editor.md)（组态层 / 数据层 / 对外接口与 Electron 多窗口模型）。

---

## 1. 文档目的

| 维度 | 说明 |
|------|------|
| **运行层** | 在总架构中位于 **组态层之下、数据层之上** 的执行侧：加载编译后的控制方案，按周期/相位调度 POU，并为功能块提供报警、趋势等系统服务。 |
| **组态工具** | 形态类比本仓库：**Electron 桌面壳 + Vue 渲染 + Rust（N-API）持久化/校验/编译编排**；产出 **控制方案程序**（及元数据），交由运行层加载执行。 |
| **本文焦点** | 归纳 C++ 运行层 **功能模块**；说明如何 **支撑现有平台的调用**；说明 **外部应用程序在运行时读取功能块数据** 的集成诉求（不绑定具体 DLL 导出表，留待实现清单）。 |

## 2. 运行层功能模块（概要）

以下与运行层分解图一致，用于与实现代码目录、对外 API 分组对齐。

| 模块 | 职责摘要 |
|------|----------|
| **程序通用外层** | 用户程序在运行时的统一封装：生命周期、与调度器的对接、与全局资源（位号、时钟等）的绑定入口。 |
| **程序调度运行** | 按 **相位、周期** 等策略驱动程序执行；保证控制回路的时间语义（确定性或尽力实时策略需在 `docs/constraints/` 约定）。 |
| **POU 管理** | 管理 **程序、功能块（FB）** 等 POU 实例：注册/注销、调用图、与组态侧 POU 标识的映射。 |
| **功能块通用外层** | FB 执行时的公共支撑：**报警、趋势** 等与系统服务的挂接；避免每个 FB 重复实现同类横切逻辑。 |
| **干扰运行** | 控制回路相关的 **扰动/注入** 能力（若产品定义为纯控制干扰或仿真场景，需在需求中写清边界与安全开关）。 |
| **报警管理** | 报警状态机、确认、优先级、与位号/POU 的关联；可为组态工具或 HMI 提供订阅或查询。 |
| **趋势管理** | 采样、缓冲、历史归档策略；为趋势控件或外部采集提供数据出口。 |

## 3. 与组态工具及编译产物的衔接

```
组态工具（Electron + Vue + Rust）
    → 工程 / 位号 / 数据类型 / 控制方案 / 报警与趋势配置
    → 编译器链路（PLCopen XML、IEC 程序、IEC2C 等，按产品裁剪）
    → 控制方案程序包（二进制/配置/清单，格式由产品约定）
    → C++ 运行层：加载、校验、调度执行
```

| 环节 | C++ 运行层侧诉求 |
|------|------------------|
| **加载** | 解析组态输出包：POU 实例表、连接关系、周期/相位表、位号映射表。 |
| **一致性** | 与组态 **数据层** 模型（类型、位号、功能块、控制方案等）字段语义一致，避免「能编不能跑」。 |
| **热更新** | 是否支持在线增量下装、是否必须停机整包替换——属产品决策，影响运行层 API 与 Rust 侧编排。 |

组态工具本体 **不替代** 运行层调度；Rust 后端可负责 **编译任务编排、产物落盘、与运行进程的版本握手**，具体进程间通信（管道、共享内存、本地 TCP、gRPC 等）在 `docs/specs/technical/` 中单独立项。

## 4. 支撑「现有平台」调用时的分工

与 [架构设想](./openplc_style_editor.md) 中的三包对应关系建议如下（实现时可调整，但边界宜保持清晰）。

| 组件 | 与运行层的关系 |
|------|----------------|
| **Electron 主进程** | 启动/停止运行层进程或服务；传递工程路径与运行参数；聚合日志与崩溃重启策略（若需要）。 |
| **Rust（N-API）** | 持久化组态数据、触发编译、将 **控制方案程序** 交付给运行层；可封装对运行层 **控制 API** 的调用（启动、暂停、单步调试等，按产品定义）。 |
| **Vue（组态 UI）** | 不直接调用 C++；通过 IPC → 主进程 / Rust 间接操作运行层；**调试/监视面板** 所需数据来自运行层经 Rust 或本地服务转发的读接口。 |

**原则**：Vue 与 C++ **不直接 FFI**；跨语言边界集中在 **Electron 主进程 + Rust**，便于统一鉴权、日志与错误模型。

## 5. 外部应用在运行时获取功能块数据

总架构 **对外接口**（WebAPI、WebSocket、TCP/IP、OPC 等）中，与「读功能块数据」强相关的集成目标可归纳为：

| 能力 | 说明 |
|------|------|
| **发现** | 外部应用能枚举或可配置地获知：实例 ID、类型名、输入/输出/可见内部变量的访问点（与组态位号或 OPC 地址映射策略一致）。 |
| **读** | 支持周期轮询或 **订阅推送**（变位、周期快照）；语义只读或带权限的写需分开定义。 |
| **与运行层模块的关系** | 读路径通常经 **接口基础层** 访问运行内存中的 FB 映像；**报警/趋势** 可走专用通道或统一数据总线，避免外部应用直接扫内存。 |
| **安全** | 鉴权、网段隔离、只读角色；防止未授权写导致扰动运行被滥用。 |

实现选型（OPC UA、自建 WebSocket、Modbus 映射等）属于 **对外接口层** 与 C++ 运行层的 **适配模块**，本文只固定 **产品能力**：**运行中，外部应用应能获取功能块相关数据**，并与组态发布的 **标签/地址模型** 一致。

---

## 6. 小结与后续文档

- C++ 运行层负责 **执行组态编译后的控制方案**，并通过 **POU/调度/FB 外层/报警/趋势** 等模块形成完整运行时。
- **现有组态平台** 通过 **Rust 编排 + Electron 进程管理** 与运行层协作；**外部应用** 通过 **对外接口** 读取（及按策略写入）功能块数据。
- 建议在 `docs/specs/technical/` 补充：**控制方案包格式**、**运行层控制与诊断 API**；在 `docs/api/` 补充：**WebAPI/WebSocket/OPC 等对外读写的具体契约**。
